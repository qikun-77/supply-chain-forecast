## 1. 计算机基础：你需要知道的最少知识

### 1.1 什么是Python？
**Python**（读作"拍森"）是一种编程语言。你可以把它理解为"给电脑下指令的文字"。就像你用中文跟人交流一样，你用Python跟电脑交流。

**类比**：Python就像你跟一个只会英文的助手沟通——你写下英文指令，他执行。Python就是你写给电脑的"指令文字"。

### 1.2 什么是终端/命令行？
**终端**（也叫命令行、CMD、Terminal、Shell）是一个黑色窗口，你在里面打字，电脑执行。

**在Windows上怎么打开**：
- 按 `Win + R`，输入 `cmd`，回车
- 或者在开始菜单搜索"命令提示符"

在这个黑色窗口里输入命令，按回车，电脑就会执行那条命令。

### 1.3 `pip` 是什么？
**pip**（读作"皮普"）= Python的"应用商店"。你想用别人写好的工具，就用pip下载。

- `pip install 包名` = 从网上下载并安装一个工具
- `pip install -r requirements.txt` = 按照购物清单，一次性安装所有需要的工具
- 本项目需要的所有工具都列在 `requirements.txt` 文件里

### 1.4 `python 文件名.py` 是什么？
这就是"运行一个Python程序"。
- `python` = 启动Python解释器（理解并执行Python代码的程序）
- `文件名.py` = 要执行的代码文件
- `.py` = Python文件的扩展名

### 1.5 什么是服务器？什么是 localhost:8501？
- **服务器**：一台持续运行的电脑，等待别人来访问它的网页
- **localhost**：就是你自己的电脑。在浏览器输入 `localhost` 就是访问你自己
- **8501**：端口号，就像一个门牌号。Streamlit这个工具默认用8501号门
- **http://localhost:8501**：在浏览器输入这个地址，就能看到你自己电脑上运行的网页

### 1.6 什么是Git和GitHub？
- **Git**：一个"版本管理"工具，记录你每次修改了什么
- **GitHub**：一个网站，可以把你用Git管理的代码上传到网上，别人也能看到
- **仓库（Repository/Repo）**：一个项目在GitHub上的"家"

## 2. 项目到底是干什么的（一句话版）

> **模拟开一个网店，每天卖东西。我们用过去3年的销售记录，预测未来30天能卖多少，然后算出每次进货多少最省钱。**

### 通俗比喻
你家楼下便利店：
- 老板每天记录卖了多少瓶水 → 这就是"**数据**"
- 他发现夏天卖得多、冬天卖得少 → 这就是"**季节性**"
- 他发现周末卖得比工作日多 → 这就是"**周模式**"
- 他发现双11打折时卖得特别多 → 这就是"**促销效应**"
- 他想预测下周该进多少货，进多了浪费钱，进少了不够卖 → 这就是"**库存优化**"
- 我们做的事情就是用数学帮老板算出最优答案

### 这个项目的完整流程

```
生成假数据 → 看图了解数据 → 5种方法预测未来 → 算最优进货量 → 做出网页展示
(模块1)      (模块2)          (模块3)           (模块4)        (模块5)
```

## 4. 核心概念：逐个击破所有专业术语

### 4.1 数据相关

| 术语 | 大白话解释 |
|------|-----------|
| **CSV** | 一种表格文件格式。就像Excel但更简单。可以双击用Excel打开，也可以用记事本打开看 |
| **日粒度** | 按天记录。比如"每天一条销售记录" |
| **时间序列** | 按时间排列的一串数字。比如"1月1日卖500件，1月2日卖480件，1月3日卖520件…"——按时间顺序排好 |
| **SKU** | Stock Keeping Unit（库存单位），就是"一种具体的商品"。比如"红色T恤L码"就是一个SKU |
| **样本/数据量** | 一共有多少条记录。本项目有1096条（365天×3年） |
| **特征** | 英文叫Feature。数据的属性，比如"日期""星期几""是不是促销日"都是特征。是用来描述数据的标签 |
| **标签** | 英文叫Label。我们要预测的那个东西，这里就是"销量" |

### 4.2 统计相关（这些概念面试必问）

| 术语 | 符号 | 大白话解释 |
|------|------|-----------|
| **均值（平均数）** | μ (读作miu) | 所有数字加起来除以个数。比如[5,10,15]三个数，均值=(5+10+15)/3=10 |
| **标准差** | σ (读作sigma) | 数字有多"散"。标准差大=数据波动大，标准差小=数据很稳定。比如[100,100,100]标准差=0（完全不波动） |
| **中位数** | - | 所有数字从小到大排，中间那个。比均值更能抵抗极端值影响 |
| **正态分布** | 钟形曲线 | 自然界最常见的分布形状。大多数数据集中在中间，两边越来越少。画出来像一口钟 |
| **变异系数** | CV = σ/μ | 标准差÷均值。用来判断数据稳不稳定。CV<0.3算稳定，>0.5算波动大 |
| **百分位数** | - | "95分位"意思是95%的数据都比这个值小 |
| **置信区间** | - | 预测不是一个数字，而是一个范围。比如"明天销量500~600件（95%把握）"。区间越窄=预测越有信心 |

**正态分布长什么样？**
```
      ╱‾‾╲
     ╱    ╲
    ╱      ╲        ← 中间最高（均值所在），两边越来越低
   ╱        ╲
──╱──────────╲──
  -3σ  -σ  μ  +σ  +3σ
```
约68%的数据落在 μ±σ 之间，约95%落在 μ±2σ 之间。

### 4.3 时间序列相关

| 术语 | 大白话解释 |
|------|-----------|
| **趋势**（Trend） | 长期来看销量是往上涨还是往下跌。比如"3年来销量从每天500涨到600"就是上涨趋势 |
| **季节性**（Seasonality） | 固定周期的规律。比如每年夏天都高、每年冬天都低；或者每周六都特别高 |
| **周期性**（Cyclicity） | 和季节性类似，但周期不固定。比如经济周期（几年一轮回，不像季节严格一年一次） |
| **噪声**（Noise） | 随机波动，没有规律的上下跳动。比如某天下暴雨销量突然跌了，这就是噪声 |
| **平稳性**（Stationarity） | 数据的均值和方差基本不随时间变化。大多数预测方法要求数据平稳 |
| **自相关**（Autocorrelation） | 今天的销量和昨天的销量有没有关系。比如"昨天卖得多→今天也可能多"=正自相关 |
| **移动平均**（Moving Average） | 取最近N天的平均值来平滑波动。7日移动平均=最近7天的平均 |
| **差分**（Differencing） | 用今天减昨天，消除趋势，让数据变平稳。ARIMA里"I"那一步做的事 |

### 4.4 预测模型相关

| 术语 | 大白话解释 |
|------|-----------|
| **模型**（Model） | 一个数学公式或算法，输入X输出Y。比如"输入前7天的销量，输出明天的销量" |
| **训练**（Training） | 让模型看历史数据，学习规律。就像学生做练习册——做得多就知道题型规律了 |
| **测试**（Testing） | 用模型没见过的数据考验它。就像考试——用新题测学生是不是真学会了 |
| **训练集**（Training Set） | 用来学习的数据。本项目用前85%的数据（约931天）做训练 |
| **测试集**（Test Set） | 用来"考试"的数据。本项目用后15%的数据（约165天）做测试 |
| **过拟合**（Overfitting） | 模型太"死记硬背"练习题了，考试换题型就不会。表现：训练集上完美，测试集上一塌糊涂 |
| **欠拟合**（Underfitting） | 模型太简单，连基本规律都没学到。表现：训练集和测试集上都很差 |
| **交叉验证**（Cross Validation） | 把数据切好几份，轮流"考"——防止模型只是凑巧在某份数据上表现好 |

### 4.5 评估指标：判断预测准不准

| 术语 | 全称 | 含义 | 怎么判断 |
|------|------|------|----------|
| **MAE** | Mean Absolute Error（平均绝对误差） | 预测值和真实值差距的平均 | 和销量同单位（件），越小越好。MAE=10说明平均每次差10件 |
| **RMSE** | Root Mean Squared Error（均方根误差） | 类似MAE但更惩罚大错误 | 对大误差特别敏感。如果某次预测差了很多，RMSE会飙升 |
| **MAPE** | Mean Absolute Percentage Error（平均绝对百分比误差） | 误差占真实值的百分比 | **我们主要看这个！** 供应链中MAPE<20%算合格，<10%算优秀 |

**举例说明三个指标**：
```
真实销量：[100, 200, 300]
预测销量：[90,  180, 330]
误差：      [-10, -20, +30]

MAE  = (10+20+30)÷3 = 20件
RMSE = √[(10²+20²+30²)÷3] = √(1400÷3) ≈ 21.6件
MAPE = [(10/100 + 20/200 + 30/300)÷3] × 100% = (0.1+0.1+0.1)÷3×100% = 10%
```

**为什么MAPE是主要指标？** 因为它是百分比，不挑单位——卖100件差10件（10%）和卖10000件差1000件（10%），MAPE都一样。可以用来比较不同商品的预测效果。

### 4.6 库存相关（供应链核心）

| 术语 | 符号 | 大白话解释 |
|------|------|-----------|
| **EOQ** | Economic Order Quantity（经济订货批量） | 每次进多少货最省钱。进太少→频繁订货费用高；进太多→库存持有费用高。EOQ就是平衡点 |
| **再订货点** | ROP（Reorder Point） | 库存降到多少时该补货了。低于这个数，马上给供应商下单 |
| **安全库存** | SS（Safety Stock） | 多备的"保命货"，防止突发情况（突然很多人买、供应商延迟送货） |
| **提前期** | L（Lead Time） | 从下单到收到货需要多少天。本项目设7天 |
| **持有成本** | H（Holding Cost） | 一件货在仓库放一年的成本（仓储费、保险费、资金占用利息）。本项目设10元/件/年 |
| **订货成本** | S（Ordering Cost） | 下一次单的固定成本（手续、运输等）。本项目设500元/次 |
| **服务水平** | - | 不缺货的概率。95%服务水平=100次补货有95次够卖，5次缺货 |
| **缺货** | Stockout | 客人要买但你没货了。缺货的代价=丢了一单生意+客户可能以后也不来了 |
| **(s,Q)策略** | - | 最经典的补货策略：库存低于s（再订货点）就下单，每次订Q（EOQ）件 |
| **ABC分类** | - | A类=重要商品（销量大/利润高，服务水准设99%）；C类=不重要（服务水准可设90%） |

### 4.7 编程相关

| 术语 | 大白话解释 |
|------|-----------|
| **函数**（Function） | 一段有名字的代码块，输入一些东西，输出一个结果。就像一个"加馅机"——输入面粉和馅，输出饺子 |
| **变量**（Variable） | 一个用来存数据的"盒子"。比如 `x = 5` 就是建了一个叫x的盒子，里面放了5 |
| **循环**（Loop） | 让电脑重复做一件事。比如"每天检查库存，如果低于ROP就下单"——重复365次 |
| **条件判断**（If-else） | "如果……就……否则……"。比如"如果库存<ROP，就下单；否则什么都不做" |
| **导入**（import） | 引入别人写好的工具。比如 `import pandas` 就是把pandas这个数据处理的工具箱拿来用 |
| **库/包**（Library/Package） | 一堆别人写好的函数，打包在一起供你使用。比如pandas就是处理表格数据的库 |

## 6. 每个模块逐行讲解

### 6.1 模块一：生成假数据 `01_data_generate.py`

**这个程序在干嘛？**
造一份假的销售记录，假装是一个网店3年的数据（2023年1月1日到2025年12月31日，共1096天）。

**为什么要造假数据？**
因为真实公司的销售数据是商业机密，拿不到。但没关系——我们自己模拟一份，原理和真实数据一样。面试官不会在意数据是不是真的，他在意你**懂不懂分析方法和业务逻辑**。

**核心代码逐行解释：**

```python
import pandas as pd
# import = 引入别人写好的工具
# pandas = Python里最常用的表格处理工具，功能类似Excel但用代码操作
# as pd = 给它起个小名叫pd，以后用pd代表pandas，少打几个字

import numpy as np
# numpy = Python里最常用的数学计算工具
# 可以做矩阵运算、生成随机数、计算三角函数等
# as np = 起小名叫np

CONFIG = {                        # CONFIG = Configuration，配置参数
    'start_date': '2023-01-01',   # 模拟数据从哪天开始
    'end_date': '2025-12-31',     # 模拟数据到哪天结束（整整3年）
    'base_demand': 500,           # 基础销量：假设平时一天卖500件
    'trend_coef': 0.15,           # 趋势系数：每天比前一天多卖0.15件
    'yearly_amplitude': 100,      # 季节性波动幅度：夏天最多比平均高100件
    'noise_std': 30,              # 随机噪声大小：每天随机±30件波动
    'promo_effect': 1.4,          # 促销效果：促销日销量是平时的1.4倍（涨40%）
    'stockout_prob': 0.02,        # 缺货概率：每天有2%的几率缺货
}
```

**造数据的数学公式（这是核心）：**

```
每日销量 = (基础销量500 + 趋势增长) × 星期几系数 × 促销系数 + 季节性波动 + 随机噪声
```

逐项解释：
- **(基础销量500 + 趋势增长)**：假设第一天卖500件，之后每天多卖0.15件。到第1000天时，这部分就有500+0.15×1000=650件
- **× 星期几系数**：周一×1.05（稍多），周日×0.9（稍少），周六×1.25（最多）。为什么？因为周末大家有空网购
- **× 促销系数**：如果是促销日（双11、618等），销量×1.4（涨40%）；平时×1.0（不变）
- **+ 季节性波动**：用正弦波模拟一年中的波动。夏天（6-8月）销量最高，冬天（12-2月）销量最低。振幅=±100件
- **+ 随机噪声**：每天还有±30件的纯随机波动，模拟现实生活中各种不可预测的因素

**生成的促销日有哪些？**
- 每月1日（模拟月初促销活动）
- 6月16-18日（模拟618大促）
- 11月10-12日（模拟双11）
- 12月12日（模拟双12）
- 总共57天促销日

**额外生成的供应链字段：**
- `lead_time`（采购提前期）：随机分配3天、5天或7天
- `cost_per_unit`（采购成本）：平均30元/件，随机波动±5元
- `selling_price`（售价）：采购成本的1.5~2.5倍

### 6.3 模块三：需求预测 `03_demand_forecast.py` ⭐ 最核心的模块

**这个程序在干嘛？**
用5种不同数学方法预测未来销量，然后对比谁最准，最后挑出冠军模型预测未来30天。

**整个流程：**
```
数据 → 切分(85%训练/15%测试) → 5种方法分别预测 → 和真实值对比 → 排名 → 最优模型预测未来30天
```

**85%训练/15%测试是什么？**
前85%的数据（约931天）给模型学习，后15%的（约165天）用来"考试"——模型没见过这165天，预测后和真实销量对比，就知道它准不准了。

---

#### 方法1：移动平均（Moving Average）— 最简单的方法

**大白话**：预测明天 = 最近7天的平均销量。

**举例**：
```
最近7天: 500, 520, 480, 510, 530, 490, 550
预测明天 = (500+520+480+510+530+490+550) ÷ 7 = 511件
```

**优点**：超级简单，任何人都能理解。作为"基准线"——如果复杂模型连这个都打不过，那就没意义了。

**缺点**：只能看过去，不会考虑趋势（涨跌）和季节变化。

---

#### 方法2：Holt-Winters 指数平滑法

**大白话**：升级版移动平均。同时追踪三个东西：①最近的销量水平、②趋势（在涨还是跌）、③季节性模式（周周期）。

**名字由来**：由统计学家 Charles Holt 和 Peter Winters 分别在1957年和1960年提出。

**核心思想**："越近的数据越重要"。给最近的数据更高权重，给远古数据更低权重。

**优点**：计算非常快，能捕捉趋势+季节性，可解释性好
**缺点**：对突变不敏感，促销这种突然事件无法提前预测

---

#### 方法3：SARIMA（季节性自回归积分滑动平均模型）

**大白话**：让数据"自己解释自己"——用前几天的销量和上周同一天的销量来预测今天。

**全称拆解**：
- **AR**（AutoRegressive，自回归）：用过去的值预测未来。比如"昨天卖500，前天卖480"→预测今天
- **I**（Integrated，差分）：如果不平稳就做差分（今天-昨天），消除趋势让数据平稳
- **MA**（Moving Average，滑动平均）：用过去的预测误差来修正当前预测
- **S**（Seasonal，季节性）：加上季节性的版本，考虑"上周今天"的影响

**参数 `(2,1,2)(1,1,1,7)` 什么意思？**
- 非季节部分 `(2,1,2)`：用前2天的自回归、做1次差分、用前2个误差修正
- 季节部分 `(1,1,1,7)`：用上周同1天、做1次季节差分、用1个季节误差、周期=7天

**优点**：经典统计方法，有扎实的数学理论，可解释
**缺点**：参数选择复杂，需要人工调参或AIC自动选参

---

#### 方法4：Facebook Prophet（"先知"模型）

**大白话**：Facebook开发的预测工具，专门为业务数据设计。你几乎不用调参数，它自动帮你处理节假日、趋势变化、季节性。

**为什么大厂爱用Prophet？**
- 自动识别周末效应（周六高、周一低）
- 自动处理节假日（618、双11可以手动加入）
- 自动发现趋势变化点（比如某时间点后销量突然暴涨）
- 结果可以拆成几部分，方便向领导汇报："趋势贡献了多少、星期几贡献了多少、节假日贡献了多少"

**Prophet的加法模型：**
```
销量 = 长期趋势 + 年的季节波动 + 周的季节波动 + 节假日影响 + 随机误差
```
就像把一道菜的每个成分都单独拎出来：这个菜咸了，是因为盐放了5克、酱油放了10毫升……

**优点**：开箱即用，自动处理各种模式，出图好看
**缺点**：对高频数据（比如每分钟一条）效率低，对极短期预测不如ARIMA

---

#### 方法5：LSTM 长短期记忆网络（神经网络/深度学习）

**大白话**：让电脑"像人一样学习"。给它看前14天的销量，让它猜第15天，猜错了就"反思调整"，循环50次后越来越准。

**什么是神经网络？**
想象一个复杂的"打分系统"：
```
输入(前14天销量) → [一层层神经元处理] → 输出(第15天预测)
                          ↑
                    通过误差反向调整连接强度
```
就像学投篮——投偏了→调整力度角度→再投→慢慢就准了。神经网络的"训练"就是这个循环调整的过程。

**LSTM有什么特别的？**
普通神经网络只看"最近几天的数据"，LSTM有一个"记忆门"——可以选择性地记住很久以前的重要信息。

**生活化类比**：
- 普通神经网络 = 金鱼的记忆，只能记住最近7秒
- LSTM = 正常人的记忆，能记住"三个月前那次促销也涨了40%"

**我们用的工具**：
- **PyTorch**：Facebook开发的深度学习框架，比TensorFlow更灵活、更"Pythonic"
- **MinMaxScaler**：把销量数据压缩到0-1之间（神经网络对数据尺度非常敏感）

**LSTM训练过程（循环50次）：**
```
第1次：前14天→预测第15天 → 差得很远 → 调整参数
第2次：前14天→预测第15天 → 好了一点 → 继续调整
...
第50次：前14天→预测第15天 → 已经相当准了 → 训练完成！
```

---

#### 5种方法对比结果

程序运行后会输出一个排名表，类似：

```
🏆 模型效果对比 (测试集)
模型                MAE      RMSE     MAPE(%)
─────────────────────────────────────────────
Prophet             12.5     15.8     2.1      ← 冠军，MAPE最小
Holt-Winters        18.3     22.1     3.0
SARIMA              20.1     24.5     3.3
LSTM                22.7     27.3     3.7
移动平均(MA-7)       35.2     41.0     5.8      ← 垫底，但作为基准线
```

**MAPE解读**：最优模型MAPE约2%-6%，说明预测非常准——每卖100件只差2-6件。

---

#### 残差诊断是什么？

**残差 = 实际销量 - 预测销量**，就是每次预测的"误差"。

**为什么要诊断残差？**
一个好的预测模型，它的误差应该：
1. **均值接近0**：不系统性地偏高或偏低（没"偏"）
2. **完全随机**：误差之间没有规律——如果今天误差是+10，明天也往往是+10，说明模型漏掉了什么
3. **和预测值无关**：不管预测多还是少，误差都差不多大

> 如果残差有规律（比如ACF图显示自相关），说明模型还可以改进——有信息没被提取出来。

### 6.4 模块四：库存优化 `04_inventory_optimize.py`

**这个程序在干嘛？**
基于需求预测的结果，用运筹学公式算出"最优进货策略"。

**核心问题**：一年要卖22.6万件货，每次进多少件最省钱？什么时候该下单？

---

#### EOQ模型详解（核心！）

**EOQ = Economic Order Quantity = 经济订货批量**

**问题背景**：两种成本在"打架"——

| 成本类型 | 怎么产生的 | 和订货量的关系 |
|----------|-----------|---------------|
| 📦 **订货成本** | 每次下单的固定费用500元 | 订得越少→订的次数越多→总订货成本越高 |
| 📦 **持有成本** | 货放在仓库里的费用10元/件/年 | 一次进得越多→平均库存越多→总持有成本越高 |

**EOQ就是这两种成本加起来最小的那个点**：

```
EOQ = √(2 × 年需求量 × 每次订货成本 ÷ 单位年持有成本)

本项目代入：
EOQ = √(2 × 225897 × 500 ÷ 10)
    = √(22589700)
    ≈ 4753件
```

**这告诉我们**：
- 每次订4753件最省钱
- 一年订47.5次（约7.7天一次）
- 订货总成本 = 47.5×500 = 23,764元
- 持有总成本 = (4753÷2)×10 = 23,764元
- **两者恰好相等——这就是最优解的特征！**

---

#### 安全库存计算

**为什么需要安全库存？**
因为现实不是完美的——可能突然很多人买（需求波动），供应商可能延迟送货。安全库存就是"保命钱/保命货"。

**公式：安全库存 = z值 × 日需求标准差 × √提前期**

```
本项目：
z值（95%服务水平）= 1.645  (从正态分布表查的)
日需求标准差 σ = 119.5
提前期 L = 7天

安全库存 = 1.645 × 119.5 × √7
        = 1.645 × 119.5 × 2.646
        ≈ 520件
```

**z值是什么？**
z值反映了你"多想避免缺货"：
- z=1.645 → 服务水平95%（100次补货有5次可能缺货）
- z=2.326 → 服务水平99%（100次只有1次缺货）
- z=1.282 → 服务水平90%（10次有1次缺货）

z值越高→安全库存越多→成本越高→缺货风险越低。这是一个权衡。

---

#### 再订货点（ROP）

**ROP = Reorder Point = 库存降到多少时就下单**

```
ROP = 提前期内期望需求 + 安全库存
    = 618.9 × 7 + 520
    = 4332 + 520
    = 4852件
```

**操作流程**：
1. 每天看库存
2. 如果库存 ≤ 4852件 → 下单一笔，订4753件
3. 等待7天→到货→库存恢复

---

#### 成本对比：优化前 vs 优化后

| 项目 | 当前策略（差） | EOQ优化策略 | 节省 |
|------|:-----------:|:---------:|------|
| 每次订货量 | 350件 | 4,753件 | - |
| 年订货次数 | 645次 | 48次 | **少了597次！** |
| 年订货成本 | 322,710元 | 23,764元 | **省29.9万** |
| 年总成本 | 7,059,038元 | 6,829,630元 | **省22.9万(3.2%)** |

**为什么能省这么多？** 原来的策略"每次只订350件"太少了，导致一年要订645次货。每次订货都有500元固定成本，645×500=32万！优化后变成每年只订48次，订货成本暴降。

> **面试官**："为什么用5种方法？Prophet不就是最好的吗？"

**你的回答**：
"因为不能先入为主地认为某种方法最好。数据分析的一个核心原则是'让数据说话'。移动平均是基准线，验证预测难度；Holt-Winters和SARIMA代表经典统计方法，可解释性强；Prophet是大厂主流工具，自动处理节假日；LSTM探索深度学习在这个场景的上限。最后根据MAPE自动选择最优。我项目中Prophet的MAPE确实最低，但如果换一种数据，可能SARIMA更好——这个对比框架的价值就在于此。"

> **面试官**："MAPE是什么？为什么不用MAE或RMSE？"

**你的回答**：
"MAPE是平均绝对百分比误差。比如预计卖100件实际卖90件，误差10件，MAPE就是10%。我选MAPE是因为它是百分比，量纲无关——卖100件差10件是10%，卖10000件差1000件也是10%。如果用MAE，后者误差是前者的100倍，无法直接比较。供应链中通常要求MAPE在20%以内，我们做到了5%以内。"

> **面试官**："EOQ模型有什么假设？你的数据满足吗？"

**你的回答**：
"EOQ假设需求稳定、提前期固定、订货成本固定、无数量折扣、不允许缺货。我们数据的需求标准差119.5，均值618.9，变异系数CV约0.19——说明需求相对稳定，EOQ基本适用。如果CV超过0.5，就需要考虑随机库存模型比如报童模型。实际落地时还要加供应商约束，但核心方法论不变。"

> **面试官**："95%服务水平怎么来的？为什么不是99%？"

**你的回答**：
"95%对应的z值是1.645。选95%是成本和服务水平的平衡——99%的z值是2.326，安全库存要多出约40%，但缺货率只从5%降到1%。除非是保命用的医疗物资，一般的电商商品95%足够了。实际中可以根据ABC分类：A类商品设为99%，B类95%，C类90%。"
